    Die Auswirkungen des neuen JDK-Release-Zyklus 02

    Wie kann man sich auf die nun doch recht schnelle Abfolge neuer JDK´s in der Entwicklung einstellen? Mit welchen Ansätzen kann man sich die Arbeit erleichtern, welche Werkzeuge werden Erleichterung verschaffen?

    Die Auswirkungen des neuen JDK-Release-Zyklus 02

    September 10, 2018
    #Java
    #JVM
    #JDK

    Wie kann man sich auf die nun doch recht schnelle Abfolge neuer JDK´s in der Entwicklung einstellen? Mit welchen Ansätzen kann man sich die Arbeit erleichtern, welche Werkzeuge werden Erleichterung verschaffen?

    Was bisher geschehen ist

    In dem letzten Teil haben wir uns angesehen wie wir ein lokales Projekt innerhalb eines Docker Containers übersetzen können und somit das Ziel JDK nicht zwingend auf der Entwicklungsplatform nativ vorhalten müssen. Allerdings hatte das bei der Verwendung von Maven den Nachteil, das der Prozess recht lange dauern kann, da alle notwendigen Abhängigkeiten in den jeweils frisch erzeugten Container geladen werden mussten.

    Heute werden wir uns verschiedene Wege ansehen wie man das beschleunigen kann.

    Und so wurde es das letzte mal gemacht

    Um nun in einem Docker Container das Projekt zu übersetzen, wurde der nachfolgende Befehl auf der Kommandzeili in dem Wurzelverzeichnis von dem Projekt ausgeführt.

    docker run -it --rm --name compileJDK10 \
        -v "$(pwd)":/usr/src/mymaven \
        -w /usr/src/mymaven \
        svenruppert/maven-3.5-jdk-zulu-10 \
        mvn clean install

    Hiermit wurde das Projektverzeichnis in dem Dockercontainer eingebunden und der gewünschte Maven Befehl ausgeführt.

    Der einfache Weg

    Kommen wir zu dem einfachsten Weg. Wenn man nun die Abhängigkeiten die in einem Projekt verwendet werden nicht jedes mal neu aus dem Internet ziehen möchte, so kann man das lokale .m2 Verzeichnis als eine Art Cache verwenden. Hierzu muss man lediglich das .m2 Verzeichnis ebenfalls in den neu erzeugten Container mit einbinden. Der in dem Container laufende Maven-Prozess sieht dann dieses Verzeichnis und wird es verwenden.

    Um das zu erreichen, müssen wir lediglich den Befehl um die folgenden Zeile erweitern. -v "path/to/m2":/root/.m2/

    Alles zusammen ergibt dann:

    docker run --rm --name compile \
    -v "path/to/m2":/root/.m2/ \ 
    -v "$(pwd)":/usr/src/mymaven \
    -w /usr/src/mymaven \
    svenruppert/maven-3.5-jdk-openjdk-11 \
    mvn clean install

    Nun gibt es aber einige Nachteile bei dieser Lösung. Gehen wir nun einmal davon aus, das wir zur selben Zeit auch lokal mit maven arbeiten werden, so kann es sein das sich die beiden Prozesse ins Gehege kommen. Das kann immer dann passieren wenn mit lokalen SNAPSHOTS gearbeitet wird. Diese Lösung kann also lediglich dann ohne Komplikationen verwendet werden, wenn man immer nur einen Prozess darauf laufen lässt.

    Cachen von Release Versionen

    Um nun die Möglichkeit zu bekommen, gleichzeitig mit verschiedenen Maven-Prozessen auf dem Rechner arbeiten zu können, muss also das .m2 Verzeichnis jeweils für den Prozess exclusiv zur Verfügung stellen. Nun wollen wir nicht beginnen, das Verzeichnis abwechselnd zu sperren. Jedoch kann man den folgenden Weg gehen. Anstelle der einfachen Bereitstellung des selben Verzeichnisses kann man mit einer Kopie arbeiten. Das Verzeichnis wird also nicht in den Container eingebunden, sondern kopiert. Nun kann der in dem Container laufende Prozess sich frei in dem Repository bewegen und Manipulationen vornehmen.

    -v "path/to/m2":/root/.m2/

    Alles zusammen ergibt dann:

    docker run --rm --name compile \
    -v "path/to/m2":/root/.m2/ \ 
    -v "$(pwd)":/usr/src/mymaven \
    -w /usr/src/mymaven \
    svenruppert/maven-3.5-jdk-openjdk-11 \
    mvn clean install

    Auch diese Methode hat einen Nachteil. Es wird immer die aktuelle Version von diesem Verzeichnis genommen. Ist es die selbe Version die auch bei der lokalen Entwicklung genommen wird, sind mit hoher Wahrscheinlichkeit auch SNAPSHOTS in dem Repository vorhanden. Das ist natürlich nicht gewünscht.

    zentraler Nexus

    Um nun die SNAPSHOTS aus dem Repository zu entfernen, muss eine reine Version aufgebaut werden. Wenn man sich genau ansieht, was am längsten dauert, so wird es bei den meisten der Download aus dem Internet sein. Hier bietet es sich an, genau diese Artefakte in einem eigenen Repository vorzuhalten. In diesem Beispiel habe ich mich für den Nexus entschieden, da er frei verfügbar ist und keine großen Konfigurationen benötigt um in Betrieb genommen zu werden.

    Aber egal welches Repositorymanager zum Einsatz kommt, der Mirror muss in der Datei settings.xml definiert werden. Nun kann man genau den selben nehmen der für die lokale Entwicklung verwendet wird. Hierzu erzeut man eine settings.xml in der die sicherheitsrelevanten Elemente entfernt worden sind und fügt diese dem Projekt hinzu. Bei dem Bau des Containers kann man dann diese Datei anstelle des .m2 Verzeichnisses hinzufügen.

    docker run --rm --name compile \
    -v "$(pwd)/_data/nexus/settings.xml":/root/.m2/settings.xml \ 
    -v "$(pwd)":/usr/src/mymaven \
    -w /usr/src/mymaven \
    svenruppert/maven-3.5-jdk-openjdk-11 \
    mvn clean install

    Nun werden alle Abhängigkeiten aus dem in dem lokalen Netzwerk vorhandenen Repository geholt. Allerdings im Vergleich zu der vorherigen Version nur die Abhängigkeiten die für diesen Build benötigt werden und ohne SNAPSHOTS. Letzteres natürlich nur wenn in dem Repository keine eingespielt worden sind.

    An der Stelle kann man nun davon ausgehen, dass der Build ausschließlich auf stabilen Versionen basiert. Die Geschwindigkeit ist abhängig von der lokalen Netzwerkgeschwindigkeit, die hoffentlich signifikant höher ist als die Internetgeschwindigkeit.

    Docker based Nexus

    Allerdings kann auch das kopieren in dem eigenen Netzwerk noch zu längeren Wartezeiten führen. Um diese zu umgehen, muss man alles auf den eigenen Rechner verlagern. Das ist dank Docker auch ein sehr einfaches Unterfangen geworden. Hier wird lediglich der Nexus in einem lokalen Docker-Container vorgehalten, und die für den Build-Prozess eingesetzten Docker Container holen sich die Maven Artefakte nun von dieser Instanz. Der Vorgang ist dann nochmals schneller, hat jedoch zusätzlich die positiven Eigenschaften das keine SNAPSHOTS enthalten sind. Um nun einen Nexus im eigenen Docker-Host bereitzustellen ist lediglich ein kurzer Befehl notwendig.

    docker run -d -p 8081:8081 --name nexus sonatype/nexus3

    Der nun laufende Nexus hat standartmäßig einen Proxy für maven central definiert und kann sofort zum Einsatz kommen. Weitere Repositories die von dem Nexus vorgehalten werden sollen, müssen allerdings von Hand defniert werden. Nun kann man einen eigenen auf dem Rechner unabhängig laufenden Nexus zur Beschleunigung verwenden.

    Dependend Builds

    Kommen wir nun zu der Anforderung, das mehr als ein Build vorhanden ist. Diese Builds allerdings in einer direkten SNAPSHOT-Abhängigkeit stehen. Nun soll also der erste Build einen SNAPSHOT erzeugen der für den darauf folgenden Build zur Verfügung gestellt werden muss. Auf jeden Fall kann/muss man nun sicherstellen, das die jeweiligen Abhängigkeiten sauber aufeinander aufbauen.

    manueller Ansatz

    Kommen wir zuerst zu dem manuellen Ansatz. Hier wird ein .m2/repository/ - Verzeichnis erzeugt, das dann bei den einzelnen Containern eingebunden wird. Ein mvn clean install wird nun die erzeugten Artefakte in diesem Verzeichnis ablegen. Wird genau dieses Verzeichnis wieder verwendet, stehen die SNAPSHOTS zur Verfügung. Letztendlich sind wir wieder bei einer der ersten Versionen die wir angedacht hatten, nur das diesmal ein frisches Verzeichnis pro Pipeline angelegt wird.

    Nexus for the build

    Wenn man an dieser Stelle die aufeinander aufbauenden Builds auf verschiedene Rechner verteilen möchte, muss man wieder mit einem Repository arbeiten das von allen beteiligten Knoten erreichbar ist. Es muss eine Nexus Instanz pro Pipeline erzeugt werden.

    vorhandene Werkzeuge

    Relativ schnell erreicht man eine Stelle, an der klar wird, dass der manuelle Aufwand zu hoch sein wird. Hier stellt sich die Frage mit welchen Werkzeugen nun gearbeitet werden kann. Es gibt hier verschiedene Ansätze die wir in den nächsten Teilen betrachten werden.

    Fazit

    Wir haben uns angesehen wie der Build-Prozess beschleunigt werden kann. Hier sind wir verschiedene Wege gegangangen. Manchmal reicht der einfache Weg aus um zum Ziel zu kommen. Nicht immer muss es eine große komplexe Infrastruktur sein. Aber selbst wenn verschiedene Teile benötigt werden, kann mittels Docker recht einfach und zügig so manche Konstruktion abgebildet werden. Auch hier gibt es wieder Potential zur Optimierung. Das werden wir in dem nächsten Teil weiter angehen.
